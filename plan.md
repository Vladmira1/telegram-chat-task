# План выполнения тестового задания "Telegram-чат"

## Что понятно сразу
*   **Цель**: Создать однопользовательский чат, похожий на Telegram, с сохранением истории сообщений.
*   **Стек**: React + TypeScript (фронтенд), Python + FastAPI (бэкенд).
*   **Основной функционал**: Отправка, получение, сохранение сообщений через REST API.

## Что требовало уточнения и поиска
*   **Дизайн**: Изучение цветовой палитры и интерфейса Telegram для создания похожего UI.
*   **Структура API**: Проектирование простых и понятных эндпоинтов (`GET /messages`, `POST /messages`).
*   **Выбор сервисов для деплоя**: Изучение и сравнение бесплатных опций. Первоначальный выбор (Vercel + Render) в процессе был скорректирован из-за технических сложностей.

## Порядок выполнения (фактический)
1.  **Настройка проекта и Git**: Создание репозитория, инициализация React (Create React App) и FastAPI приложений.
2.  **Разработка бэкенда (FastAPI)**: Создание API, настройка CORS, интеграция SQLite для хранения сообщений.
3.  **Разработка фронтенда (React)**: Создание компонентов чата, реализация логики отправки/получения, стилизация под Telegram.
4.  **Локальная интеграция и отладка**: Подключение фронтенда к локальному бэкенду, исправление ошибок типов и CORS.
5.  **Деплой бэкенда**: Публикация FastAPI-приложения на **Render.com**. Решение проблем с `requirements.txt` и командой запуска.
6.  **Деплой фронтенда**: Публикация React-приложения на **Netlify**. Решение проблем с путями сборки (`homepage` в `package.json`) и настройками CI.
7.  **Фиксация проблем интеграции**: Обнаружение и попытка решения проблемы с CORS-политикой между Netlify и Render. Внедрение временного CORS-прокси для демонстрации работоспособности.
8.  **Документация**: Написание итогового плана и саморевью.

## Принятые решения и их обоснование
*   **FastAPI для бэкенда**: Быстрый старт, встроенная автоматическая документация (Swagger/OpenAPI), асинхронная поддержка.
*   **SQLite для хранения данных**: Простота, не требует отдельного сервера, соответствует требованию "сохранять сообщения". Показывает умение работать с БД, что является плюсом.
*   **Простой API дизайн**: Эндпоинты `GET /messages` и `POST /messages` полностью покрывают требуемый функционал, их легко тестировать и понимать.
*   **Деплой на Render и Netlify**: Бесплатные, популярные сервисы, хорошо интегрированные с GitHub. Render удобен для Python-приложений, Netlify — для статических React-сайтов.
*   **Использование CORS-прокси (временно)**: Решение для демонстрации работоспособности полного цикла "фронтенд-бэкенд" до окончательной настройки CORS на продакшн-сервере.
